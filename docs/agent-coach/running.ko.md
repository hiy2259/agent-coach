# 루프 실행하기

> agent-coach는 Claude Code 스킬이므로 **자연어로 지시해서** 움직입니다. 사용자가 할 일은
> 매 반복을 일일이 승인하는 것이 아니라, 시험지(골든셋)와 설정을 설계하는 것입니다.
> 정상적인 사용에서는 스크립트를 직접 호출할 일이 없습니다. 스킬이 알아서 호출합니다.
> 되돌릴 수 없는 행동은 맨 마지막에 딱 하나 있습니다: 스테이징된 결과를 **커밋하거나
> 되돌리는 것**입니다.

이 페이지는 실행을 처음부터 끝까지 따라갑니다: 시작 방법, 스킬이 하는 일, 세 가지 실전
시나리오, 중단된 실행 재개, 골든셋이 아직 없을 때의 콜드 스타트 경로, 그리고 실행 비용.
*시험지*에 대해서는 [`golden-set.ko.md`](./golden-set.ko.md)를, *설정*에 대해서는
[`run-config.ko.md`](./run-config.ko.md)를 보세요.

---

## 빠른 시작

전형적인 시작은 문장 하나면 됩니다:

> "`./prompts/summarizer.md`를 `golden-set.json`의 케이스들에서 점수가 더 잘 나오도록
> 튜닝해줘(train과 held-out은 이미 표시돼 있어). 단, **무엇이든 바꾸기 전에 먼저 측정**하고,
> 정말로 도움이 될 때만 변경을 유지해. 과적합은 하지 마. 설정은 `run-config.json`에 있어."

이 스킬은 "예시 케이스에 맞춰 프롬프트를 최적화/튜닝/강화해 달라", "실패율을 낮춰 달라",
"eval·골든셋 기반 개선 루프를 만들어 달라" 같은 요청에 자동으로 실행됩니다. "루프"라는
단어를 한 번도 쓰지 않아도 됩니다. 실행되면 다음 순서로 진행합니다:

1. **사전 점검**으로 설정을 검증하고, 판정을 내리는 핵심 코드가 정상 동작하는지 테스트로
   확인합니다.
2. 무언가를 제안하기 *전에*, train + held-out에서 **기준 점수(베이스라인)**를 잽니다.
3. 코드가 정한 통과 조건(게이트)이 걸린 루프를 돌립니다. 턴마다 변경은 한 가지뿐이고,
   모든 편집은 **작업용 사본에서만** 일어납니다.
4. 마지막에 결과를 **묶음(batch)**으로 제시합니다: 처음과 최종 후보를 비교한 디프(diff),
   점수 추이가 담긴 `history.jsonl`, 시도했다가 버린 기록인 `failure-log.jsonl`. 사용자는
   이것을 보고 **커밋**(실제 파일에 대한 처음이자 유일한 쓰기)하거나 **되돌립니다**.

**이 루프는 반(半)자율적입니다. 변경 하나하나에 승인을 요청하지 *않습니다*.** 사용자의
일은 시험지와 설정을 설계하는 것이고, 턴은 알아서 돌아갑니다. 다만 마지막 품질 확인,
즉 개선된 프롬프트를 실제로 *써 보는 일*은 사용자의 몫으로 남습니다. 골든셋 점수 통과는
필요조건이지 충분조건이 아닙니다.

### 판정 코드 직접 검증하기

실행을 신뢰하기 전에, 되돌릴 수 없는 결정을 전부 내리는 그 코드를 직접 검증해 볼 수
있습니다:

```bash
python3 skills/agent-coach/scripts/tests/run.py          # 테스트 219개, 표준 라이브러리만 사용
# 또는:  python3 -m pytest skills/agent-coach/scripts/tests/
```

모든 스크립트는 JSON 입력을 **stdin**(또는 파일 경로 인자)으로 받습니다. 명령줄 인자에
JSON을 직접 넣는 방식이 아닙니다:

```bash
printf '%s' '<json>' | python3 skills/agent-coach/scripts/score_compare.py
```

(스크립트는 저장소 루트에서 실행하세요. 경로가 이 `docs/` 페이지가 아니라 프로젝트 루트
기준이기 때문입니다.)

## 시나리오별 사용법

아래 세 가지 시나리오는 스킬 자체를 처음부터 끝까지 검사하는 자체 평가
([`../../skills/agent-coach/evals/evals.json`](../../skills/agent-coach/evals/evals.json))와
짝을 이룹니다.

### 시나리오 1 — 웜 스타트 (골든셋이 이미 있는 경우)

**가진 것:** 대상 파일, 그리고 `train`/`heldout`이 이미 표시된 `golden-set.json`.

> "`./prompts/summarizer.md`를 `golden-set.json` 케이스에서 점수가 더 잘 나오게 튜닝해줘.
> 단, **무엇이든 바꾸기 전에 먼저 측정**하고, 정말 도움이 될 때만 유지해. 과적합 금지.
> 설정은 `run-config.json`."

**무슨 일이 일어나나:** 설정 검증 → train + held-out에서 베이스라인 측정 → 게이트가 걸린
루프 실행(턴당 변경 한 가지, 작업용 사본에서만) → 노이즈를 이기고 확인 재실행까지 통과한
변경만 병합 → 디프 + `history.jsonl` + `failure-log.jsonl`을 받아 **커밋 또는 되돌리기**
결정. 커밋하기 전까지 실제 파일은 그대로입니다.

### 시나리오 2 — 콜드 스타트 (골든셋이 아직 없는 경우)

**가진 것:** 대상 파일과 (있다면) 가공되지 않은 로그 더미. **정답이 표시된 테스트셋은
없습니다.**

> "`classify.md`가 내 고객지원 티켓 분류기인데 자꾸 오분류해. 더 좋게 만들어줘. 테스트셋은
> 없고 `seed-logs.txt`에 원시 수신함 줄들만 있어. 설정은 `run-config.json`."

**무슨 일이 일어나나:** 스킬이 골든셋이 없음을 감지합니다. 이때 스스로 하나 만들어 자기
채점을 하지는 **않습니다**. 대신 **Bootstrapper(초안)**가 입력 *후보*를 초안하고, 실제로
어디서 틀리는지 드러나도록 대상을 한 번 실행한 뒤, 루프는 **사람 확인 지점에서 멈춥니다**.
입력을 골라내고 예/아니오 채점 기준을 쓰는 일은 사용자에게 요청하며, 이때 크기
게이트(`train ≥ 5`, `heldout ≥ 3`)도 알려 줍니다. 사용자가 정리를 마친 뒤에야 최적화
루프가 시작됩니다. 전체 경로는 아래
[콜드 스타트](#콜드-스타트-골든셋이-아직-없을-때)에 자세히 있습니다.

### 시나리오 3 — 과적합 방어 (일반화를 깨뜨리는 솔깃한 변경)

**가진 것:** 대상 파일과 골든셋, 그리고 "그냥 항상 X 하게 만들면 되잖아?" 같은 손쉬운 수정
아이디어.

> "채용공고 필드 추출기 `extract.md`를 `golden-set.json`에 맞춰 튜닝해줘. 제일 큰 문제는
> `salary_range`를 너무 자주 null로 두는 거야. 이 값이 채워졌으면 좋겠어. 설정은
> `run-config.json`."

**무슨 일이 일어나나:** 누구나 떠올릴 법한 "급여를 항상 채워 넣기" 변경은 **train** 점수를
올립니다. 하지만 급여가 명시되지 않은 공고들이 모여 있는 **held-out**에서는 점수를
무너뜨립니다. 그래서 코드가 **HALT**를 반환하고, 그 변경은 대상 파일에 반영되지 않습니다.
중단된 시도는 `candidate_input`(다음 골든셋 버전에 추가할 만한 케이스)과 함께 기록으로
남습니다. 결과적으로, 프롬프트가 없는 값을 지어내도록(환각) 학습될 뻔한 것을 막아 낸
것입니다.

## 중단된 실행 재개

모든 실행은 재개할 수 있습니다. 실행이 중단되면 `state.json`에 현재 턴과 단계가 기록되어
있고, `resume.py`가 마지막으로 완료된 단계부터 다시 진입합니다. 이 재진입은
**멱등**합니다. 즉, 같은 단계를 다시 밟아도 병합이 두 번 적용되는 일은 없습니다.

> "`summarizer.md`에 대한 agent-coach 실행, 멈춘 지점부터 이어서 해줘."

샘플 `state.json`(그리고 함께 있는 `history.jsonl`, `failure-log.jsonl`)이
[`../../examples/agent-coach/ko/loop-state/`](../../examples/agent-coach/ko/loop-state)에
있으니, 실행을 재개할 때 쓰이는 상태 기록에 정확히 무엇이 담기는지 직접 볼 수 있습니다.

## 콜드 스타트: 골든셋이 아직 없을 때

골든셋이 없다고 해서 루프가 스스로 하나 지어내 그것으로 채점하지는 **않습니다**. 그것은
자기가 낸 시험을 자기가 치는 꼴이기 때문입니다. "골든셋 없음" 상태는 *코드로*
판정됩니다(`split_goldenset.py op=state`가 `missing`/`empty`를 반환). 그다음 순서는
이렇습니다:

1. **시작 입력 모으기.** 로그가 있으면 거기서 입력을 가져오고, 필요하면 **Bootstrapper**에게
   입력 *후보* 초안을 만들게 합니다. 로그가 있어도 그 내용이 쉽거나 순탄한 경우뿐이라면
   Bootstrapper를 함께 돌리세요. 대상이 전부 통과해 버리는 쉬운 셋으로는 약점을 하나도
   찾아내지 못하기 때문입니다.
2. **실패 드러내기.** 후보 입력에 대해 대상을 한 번 실행(Runner)해서, 실제로 어디서
   틀리는지 확인합니다.
3. **사람이 골라냅니다.** 입력을 승인하거나 쳐내고, 빠져 있는 어려운 케이스를 추가하는 것은
   *사용자*입니다. 채점 기준만이 아니라 입력 선택까지 사용자가 직접 맡아야, 시험 문제와
   응시자가 같은 사각지대를 공유하는 일을 막을 수 있습니다.
4. **사람이 채점 기준을 씁니다.** 예/아니오 기준(입력당 5~7개)을 *사용자가* 작성합니다.
5. **분할.** `split_goldenset.py`가 정리가 끝난 *뒤에* train/held-out을 나눕니다. **가장
   현실적인** 항목을 held-out에 배치하고, 분할 결과를 해시값으로 잠가 이후에 바뀌지 않게
   확정합니다(동결).
6. **크기 게이트.** `train < 5` 또는 `heldout < 3`이면 거부하거나 강하게 경고합니다.

루프는 이 확인 지점에서 멈춰서 입력 승인과 채점 기준 작성을 요청합니다. 셋을 조용히
확정해 놓고 거기에 최적화하는 일은 하지 않습니다. 셋에 *무엇을* 담아야 하는지는 별도의
페이지에서 다룹니다: [`golden-set.ko.md`](./golden-set.ko.md)를 보세요. Bootstrapper가
무엇을 하고 무엇을 하지 않는지는
[`../../skills/agent-coach/agents/bootstrapper.md`](../../skills/agent-coach/agents/bootstrapper.md)에
있습니다.

## 한 번의 실행 비용

한 턴의 대략적인 구성:

```
Runner (현재 + 후보 검증 + 확인) × 셋 크기
  + Grader (동일) + Proposer (1)
  ≈ |train| = 5, |held| = 3일 때, 병합이 일어난(promote) 턴에서 모델 호출 약 65회
    (확인 단계가 후보와 현재/베이스라인 프롬프트를 둘 다 재실행하기 때문)
  ≈ 병합이 없는 턴은 약 33회 (DISCARD/HALT는 확인 단계 생략)
  + k_calib회의 보정 실행 (콜드 스타트 때 1회)
```

예산은 **권장 사항이 아니라 강제 제약**입니다: `run-config.json`에
`budget.max_usd_total`(총액)과 `budget.max_usd_per_turn`(턴당)을 설정하세요. 총비용은
`n_turns`와 골든셋 크기에 비례합니다. 루프는 `state.json`의 `budget_spent_usd`로 지출을
추적하다가, 상한에 닿으면 멈춥니다.
