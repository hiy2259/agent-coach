# agent-coach 동작 원리 — 설계와 안전장치

> agent-coach의 측정 루프가 어떻게 움직이고 왜 그렇게 설계되었는지를 다룹니다: 이 스킬이 필요한 이유, 무엇이 다른가, 루프 한 턴의 진행 과정, 병합 게이트의 판정 방식, 일곱 가지 안전 규칙, 그리고 번들 스크립트.

## 왜 이 스킬이 필요한가

"자기개선(self-improvement)"은 멋지게 들리지만, 측정 없이 하는 순간 함정이 됩니다. 모델이 프롬프트를 고치고, 스스로 "더 좋아졌다"고 선언하고, 그런 편집이 쌓이면서 프롬프트는 아무 근거 없이 조금씩 바뀌어 갑니다. 실제로는 나빠지고 있어도 아무도 알아채지 못합니다. 이렇게 눈에 띄지 않게 쌓이는 변질 속에는 구체적인 실패 유형 세 가지가 숨어 있습니다:

1. **노이즈를 진보로 착각합니다.** 실사용 temperature에서는 같은 프롬프트라도 실행할 때마다 출력이 조금씩 달라지고, 점수도 그만큼 출렁입니다. "train 점수 +1"은 순전히 운일 수 있습니다. 운 좋게 나온 점수를 근거로 변경을 유지하다 보면, 점수는 슬금슬금 오르는데 프롬프트 자체는 오히려 *나빠집니다*. 우연을 텍스트에 새겨 넣은 셈이기 때문입니다.
2. **과적합(overfitting).** 어떤 변경은 튜닝에 쓴 바로 그 케이스들에서만 점수가 오릅니다. 그 케이스들의 특징을 외워 버린 것이라서, 그 밖의 상황에서는 조용히 나빠집니다. 전형적인 "테스트셋에서는 98%, 운영에서는 고장" 함정입니다.
3. **자기가 낸 시험을 자기가 채점합니다.** 방금 변경을 만든 모델에게 "네 변경이 도움이 됐어?"라고 물으면, "그렇다"고 답하는 쪽으로 이미 기울어 있습니다. 선수가 자기 경기의 심판을 봐서는 안 됩니다.

agent-coach는 이 세 가지 실패 유형을 각각 **기계적으로**, 즉 선의가 아니라 코드와 절차로 차단하도록 설계되었습니다. 그래서 "더 낫다"가 의견이 아니라 **측정된 사실**이 됩니다.

## 무엇이 다른가

| 흔한 "AI가 프롬프트를 개선해 주는" 도구 | agent-coach |
|---|---|
| 변경이 도움이 됐는지 모델이 판단 | **모델이 아닌 고정된 코드**가 정해진 부등식으로 모든 병합을 판정 |
| 점수 하나를 놓고 직접 최적화 | **train + held-out 분리**: held-out이 일반화를 지키고, 과적합이면 **HALT**(중단) |
| "+1"이면 무조건 수용 | 향상이 **측정 노이즈를 넘어야** 하고, 확인용 재실행에서도 유지되어야 함 |
| 같은 모델이 제안하고 채점 | **네 역할 분리** (제안 ≠ 채점 ≠ 실행 ≠ 초안) |
| 실제 파일을 바로 수정 | **스테이징 전용**: 사용자가 커밋하기 전까지 실제 파일은 그대로 |
| 규칙이 쌓이기만 함 | 3턴마다 **빼기 턴**: 규칙 하나를 지워 보고 다시 측정 |
| 이미 실패한 아이디어를 반복 | 제안 역할이 **실패 로그**를 먼저 읽어, 확인된 막다른 길은 다시 시도하지 않음 |

## 동작 방식

### 네 가지 역할

가장 중요한 구조 규칙 하나: **변경을 제안한 모델은 그것을 채점하지 않고, 그 둘 중 어느 쪽도 대상을 실행하는 모델이 아닙니다.** 각 역할은 [`skills/agent-coach/agents/`](../../skills/agent-coach/agents) 아래에 자기만의 프롬프트를 가진 별도의 Claude Code 서브에이전트로 돌아갑니다. 각자 깨끗한 컨텍스트에서 시작하므로, 역할 분리가 이름뿐이 아니라 실제로 지켜집니다.

| 역할 | 하는 일 | 비고 |
|---|---|---|
| **Runner(실행)** | 골든셋 입력에 대해 대상 프롬프트를 실행해 출력을 만듭니다 | 격리 실행: 쓰기·네트워크·셸 접근이 없고, 골든셋 입력만 읽습니다. 이 격리는 **보안 경계**이기도 합니다. 어떤 대상 프롬프트가 들어오더라도 호스트를 장악할 수 없어야 하기 때문입니다. 실제 운영에서 쓰는 모델·temperature 그대로 실행하므로, 측정되는 노이즈가 실사용에서 겪을 노이즈와 같습니다. |
| **Grader(채점)** | 출력을 채점 기준(루브릭)에 따라 기준마다 yes/no로 채점합니다 | **temperature 0** + 버전 고정으로 실행되어, 채점 과정은 무작위성을 전혀 더하지 않습니다. |
| **Proposer(제안)** | 턴마다 정확히 **한 가지** 변경만 제안합니다 | 먼저 `failure-log.jsonl`을 읽어, 이미 실패한 방향은 다시 제안하지 않습니다. |
| **Bootstrapper(초안)** | 콜드 스타트(골든셋이 아직 없을 때)에 입력 *후보*를 초안합니다 | 반드시 Grader와 다른 모델이어야 합니다. |

### 루프 한 턴

루프는 최대 **N**턴(기본 10) 돕니다. 각 턴은 많아야 한 가지만 바꾸고, 그 결과는 코드가 판정합니다:

```
① 실행      Runner가 현재 대상을 모든 train + held-out 입력에 대해 실행
② 채점      Grader가 출력을 채점 → train_before, heldout_before
③ 제안      Proposer가 변경 1건 제안 {target_id, before, after, rationale, kind}
④ 검증      verify_change.py: `before` 텍스트가 정확히 한 곳에만 일치하는지,
            수정이 충분히 작고 국소적인지 확인. 아니면 이번 턴을 거부
            ("정확히 한 곳만 고친다"를 코드로 강제)
⑤ 적용      apply_change.py: 스테이징 사본(prompt.candidate.md)에만 기록
            (실제 대상 파일은 건드리지 않음)
⑥ 재실행    Runner가 후보(수정본)를 모든 입력에 대해 실행
⑦ 재채점    Grader가 다시 채점 → train_after, heldout_after
⑧ 비교      score_compare.py가 코드로 판정: MERGE / DISCARD / HALT
⑨ 확인      MERGE일 때만: 후보와 현재(베이스라인) 프롬프트 둘 다 한 번 더 실행 + 채점.
            새로 잰 베이스라인과 비교해도 향상이 유지되어야 함
⑩ 기록      MERGE   → 후보를 현재 버전으로 승격 + history.jsonl에 한 줄 기록
            DISCARD → failure-log.jsonl에 기록(+ candidate_input); 실제 파일은 그대로
            HALT    → 실행 중단 + 경고 + failure-log.jsonl 기록 (result: halted)
```

**확인 단계(⑨)는 왜 재채점이 아니라 재실행일까요?** Grader는 temperature 0으로 돌기 때문에, 같은 텍스트를 두 번 채점하면 완전히 같은 점수가 나옵니다. 그래서 재채점만으로는 아무것도 확인되지 않습니다. 진짜 무작위성은 실행할 때마다 다른 출력을 내는 **Runner** 쪽에서 나옵니다. 그래서 병합 확인은 대상을 다시 실행해 새 출력을 채점하는 방식으로 이루어집니다. 이때 **후보와 현재(베이스라인) 프롬프트를 둘 다** 다시 실행하고, 새로 잰 베이스라인(`train_b2` / `held_b2`)을 기준으로 게이트를 다시 검사합니다. 처음 쟀던 베이스라인을 그대로 재사용하면 두 검사가 같은 우연을 공유하게 되어, 요행을 걸러내는 확인 단계의 힘이 대략 절반으로 줄기 때문입니다. 새 베이스라인과 비교했을 때 향상이 사라졌다면, 그것은 처음부터 노이즈였던 것입니다.

### 병합 게이트 (핵심)

판정은 모델이 아니라 스크립트 [`score_compare.py`](../../skills/agent-coach/scripts/score_compare.py)가 내립니다. 변경은 **다음 조건이 전부 성립할 때만 MERGE 됩니다:**

```
train_after   >  train_before                      (엄격히 양(+)의 향상. +0.0 동점은 절대 병합되지 않음)
train_after   ≥  train_before   + eps_train        (train에서 측정 노이즈를 넘는 향상)
heldout_after ≥  heldout_before − eps_heldout       (held-out에서 실질적인 퇴행 없음)
그리고 확인 재실행(⑨) 후에도 향상이 유지될 것
```

그 외의 경우:

- train은 오르는데 held-out이 `eps_heldout`보다 **더 많이** 떨어지면 → **HALT**. 변경이 train 케이스를 외워 버려 일반화가 깨진 것, 즉 과적합이 적발된 것입니다. HALT가 나오면 실행은 그대로 끝납니다.
- 그 밖의 모든 경우 → **DISCARD**. 실질적인 향상이 입증되지 않았으므로 그 변경은 버립니다.

허용 오차 `eps_train` / `eps_heldout`은 **측정 노이즈의 크기**이며, [`calibrate_noise.py`](../../skills/agent-coach/scripts/calibrate_noise.py)가 이렇게 잽니다: 같은 고정 입력에 대해 Runner를 `k_calib`번(5회 이상 권장) 반복 실행하고, 매번 채점해서, 분할(train/held-out)별로 점수가 얼마나 퍼지는지 기록합니다. 결과값에는 작은 양수 `min_eps`라는 하한을 두어, +0.0 동점이 진보처럼 보이는 일을 막습니다. held-out 허용 오차는 **대칭**입니다. 병합 판정과 HALT 판정에 같은 `eps_heldout`을 쓰기 때문에, held-out 점수의 평범한 출렁임이 억울한 HALT를 일으키지 않습니다.

**게이트가 애초에 통과 가능한 상태인지부터 확인하세요.** 베이스라인 점수를 `calibrate_noise.py`에 넘기면 됩니다. `gate_satisfiable: false`가 반환된다면, `eps_train ≥ 1 − baseline_train`이어서 *어떤* 변경도 게이트를 넘을 수 없는 상태라는 뜻입니다(베이스라인이 이미 ≈ 1.0인 포화 상태도 여기에 포함됩니다). 이때 루프는 N턴을 낭비하지 않고 **멈춰서 경고를 보고합니다**. 해결책은 더 크거나 더 어려운 골든셋, 또는 더 큰 `k_calib`이지, 제안을 더 시켜 보는 것이 아닙니다.

### 빼기(subtraction) 턴

본능에 맡겨 두면, 프롬프트를 고치는 사람은 누구나 규칙을 **추가**만 하게 됩니다. 그렇게 단서 조항이 쌓이면 텍스트는 비대해지고 결국 자기모순에 빠집니다. 그래서 루프는 **3턴마다** 질문을 뒤집어 "무엇을 더할까?"가 아니라 **"무엇을 뺄 수 있을까?"**를 묻습니다. 효과가 없어 보이는 규칙 하나를 지워 보고 다시 측정하는 것입니다.

- `train_after ≥ train_before − eps_train` **이고** `heldout_after ≥ heldout_before − eps_heldout`이면 제거를 유지합니다. 측정 가능한 손해 없이 프롬프트가 더 간결해진 것이므로, `SUB_KEEP`으로 기록하고 '진전 없는 턴' 카운터를 0으로 되돌립니다.
- 그렇지 않으면 규칙을 되살리고 `SUB_DROP`으로 기록하며, 카운터는 1 올라갑니다.

(`SUB_DROP`은 *제거*를 되돌린 것이고, `DISCARD`는 *추가*를 되돌린 것입니다. 서로 다른 결과입니다.)

### 정지 조건

루프는 다음 중 가장 먼저 도달하는 조건에서 멈춥니다: **N턴 도달** · **연속 K턴 동안 진전 없음**(기본 3; `MERGE`/`SUB_KEEP`이 나오면 카운터 초기화) · **예산 초과**(턴당 또는 총액) · **만점 달성** · **HALT**.

기대한 점수에 못 미친 채 멈추는 **정체(plateau)는 실패가 아니라 정보입니다.** 지시문 텍스트가 지금의 모델과 도구로 낼 수 있는 상한에 닿았다는 뜻이기 때문입니다. 더 올라가려면 문구가 아니라 모델을 바꾸거나 도구를 추가해야 합니다. 루프는 헛돌지 않고 정체를 있는 그대로 보고합니다.

## 일곱 가지 안전 규칙 (S1–S7)

이 규칙들이 이 스킬이 존재하는 이유입니다. 각 규칙은 "자기개선"이 측정 없는 마구잡이 변경으로 조용히 미끄러지는 경로를 하나씩 막습니다. 전체 원문: [`references/safety-invariants.md`](../../skills/agent-coach/references/safety-invariants.md).

| # | 규칙 | 막는 것 | 강제하는 코드 |
|---|---|---|---|
| **S1** | held-out 분리 + 과적합 시 **HALT** (대칭 `eps_heldout` 허용 오차) | 과적합된 변경이 병합되는 것 | `split_goldenset.py`, `score_compare.py` |
| **S2** | **병합은 코드가 결정** (모델의 "더 낫다"로는 불가; 향상은 엄격한 `>` 필요) | 희망 섞인 자기 평가 | `score_compare.py` |
| **S3** | **기계적으로 검증된 단일 변경** (`before` 유일 일치 + 국소성 상한) | 모호하거나 여러 개가 뒤섞인 편집 | `verify_change.py` (`apply_change.py`가 재확인) |
| **S4** | **스테이징** (사람이 커밋하기 전까지 실제 파일 불변) | 실제 프롬프트가 몰래 손상되는 것 | `apply_change.py` |
| **S5** | **케이스 출처는 사람이 관리** + 최소 크기 게이트 (`train ≥ 5`, `heldout ≥ 3`) | 자가 채점 시험, 너무 작은 시험 | Bootstrapper ≠ Grader, 사람이 직접 선별, `split_goldenset.py` |
| **S6** | **실패 로그**를 제안 역할이 다시 읽음 | 막다른 길 반복, 진화가 멈춘 골든셋 | `failure-log.jsonl`, `agents/proposer.md` |
| **S7** | **실제 Runner 분산으로 잰 노이즈 허용 오차** + 확인 재실행 | 노이즈를 진보로 착각하는 것 | `calibrate_noise.py`, 확인 단계 |

일곱 중 하나만 빠져도 루프는 의미 없는 점수를 기어오를 수 있습니다. 일곱이 모두 지켜질 때 비로소 `MERGE`가 진짜 의미를 갖게 됩니다.

## 번들 스크립트

되돌릴 수 없는 결정은 전부 코드가 내립니다. 모델은 텍스트를 생성하고 채점하는 일만 합니다. 모든 스크립트는 [`skills/agent-coach/scripts/`](../../skills/agent-coach/scripts) 아래에 있습니다.

| 단계 | 스크립트 | 역할 |
|---|---|---|
| 사전 점검 | `validate_config.py` | 실행 설정 검증: 역할 분리, Grader temperature 0 |
| ④ | `verify_change.py` | 제안이 정확히 하나의 국소 변경인지 검증 (`before` 유일 일치 + 국소성 상한) |
