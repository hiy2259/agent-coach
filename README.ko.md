# Loop — 측정 기반 프롬프트 개선

> **프롬프트·스킬·지시문 파일을 감(感)이 아니라 *측정*으로 개선하는 [Claude Code](https://docs.claude.com/en/docs/claude-code) 스킬 2종.**

*다른 언어로 보기: [English](./README.md)*

이 저장소에는 **짝을 이루어 동작하는 Claude Code 스킬 두 개**가 들어 있습니다(둘 다 [`skills/`](./skills) 아래에 있습니다). 두 스킬은 **"측정 없는 진화는 없다"**라는 한 가지 원칙을 공유합니다. 측정으로 "도움이 됐다"고 확인되지 않은 변경은 유지하지 않는다는 뜻입니다. 그리고 두 스킬은 서로 이어서 쓰도록 설계되어 있습니다:

| 스킬 | 하는 일 | 언제 쓰나 |
|---|---|---|
| [**agent-coach**](#agent-coach) | 대상 프롬프트/스킬/지시문 파일을 조금씩, 측정해 가며 개선합니다: 점수를 재고 → 한 가지만 바꾸고 → 다시 재서 → 점수가 정말 올랐을 때만 그 변경을 유지합니다. | 골든셋이 이미 있고, 그 기준으로 프롬프트를 개선하고 싶을 때 |
| [**golden-set-drafter**](#golden-set-drafter) | 골든셋이 없을 때 agent-coach가 쓸 골든셋 초안을 대신 만들어 줍니다. 단, held-out 채점 기준만은 **사용자가 직접** 써야 하며, 그 지점에서 멈춥니다. | 프롬프트를 개선하고 싶은데 **골든셋이 아직 없을 때** |

**골든셋이란?** 프롬프트를 채점하는 데 쓰는 작은 시험 문제집입니다. 문제(케이스) 하나는 "실제로 들어올 법한 입력"과 "예/아니오로 답할 수 있는 채점 질문 목록"(예: "답변이 X를 언급했는가?")으로 이루어집니다. 그중 **held-out**이라고 부르는 일부 문제는 따로 떼어 두고, 개선 작업에는 절대 쓰지 않습니다. 개선 결과가 "한 번도 연습하지 않은 문제"에서도 통하는지 확인하는 검증 전용이기 때문입니다.

골든셋이 아직 없다면 **golden-set-drafter**를 먼저 돌려 초안을 만들고, 그 결과를 **agent-coach**에 넘기면 됩니다.

## agent-coach

agent-coach는 코치가 선수를 훈련시키듯 프롬프트를 개선합니다: **점수를 잰다 → 딱 한 가지만 바꾼다 → 다시 잰다 → 점수가 정말 올랐을 때만 유지한다.** 감으로 고치는 일은 없습니다. 변경을 유지할지 버릴지는 모델의 의견이 아니라 **정해진 코드**가 판단합니다.

agent-coach가 `MERGE`(변경 유지)라고 보고했다면, 구체적으로 다음이 모두 확인되었다는 뜻입니다:

- 그 변경으로 train(훈련) 점수가 측정 노이즈(우연한 점수 출렁임)보다 크게 올랐습니다.
- held-out(검증) 점수는 나빠지지 않았습니다(일반화 확인).
- 한 번 더 재실행해도 그 향상이 그대로 유지되었습니다.
- 그동안 실제 파일은 한 글자도 바뀌지 않았습니다. 마지막에 사용자가 커밋해야 비로소 반영됩니다.

"AI에게 프롬프트를 고쳐 달라고 하기"와 무엇이 다른가:

- **유지/폐기를 코드가 결정합니다.** 모델의 "더 나아 보인다"가 아니라 고정된 부등식으로 판단합니다.
- **train / held-out을 분리합니다.** 변경은 train 문제로 다듬고, held-out 문제로 검증합니다. train 점수는 오르는데 held-out이 무너지면 과적합이므로 루프가 **HALT**(중단)합니다.
- **점수 향상은 측정 노이즈를 넘어야 하고**, 확인용 재실행에서도 유지되어야 합니다. 운 좋게 나온 +1은 걸러집니다.
- **네 가지 역할을 분리합니다.** 변경을 제안한 모델이 채점까지 하는 일은 없습니다(제안 ≠ 채점 ≠ 실행 ≠ 초안).
- **스테이징 전용입니다.** 모든 편집은 작업용 사본에서만 일어나고, 실제 파일은 사용자가 커밋하기 전까지 그대로입니다.

### 사용법

자연어로 지시하면 됩니다. 스크립트는 스킬이 알아서 실행합니다. 전형적인 시작 문장:

> "`./prompts/summarizer.md`를 `golden-set.json`의 케이스들에서 점수가 더 잘 나오도록 튜닝해줘(train과 held-out은 이미 표시돼 있어). 단, **무엇이든 바꾸기 전에 먼저 측정**하고, 정말로 도움이 될 때만 유지해. 과적합은 하지 마. 설정은 `run-config.json`에 있어."

그러면 스킬이 설정을 검사하고, train + held-out에서 **기준 점수(베이스라인)**를 먼저 잰 다음, 턴마다 한 가지 변경만 시도하는 루프(모든 변경은 사본에서만)를 돌리고, 마지막에 결과 묶음을 건네줍니다. 사용자는 그것을 **커밋하거나 되돌리기만** 하면 됩니다. 실제 파일이 바뀌는 순간은 커밋 그 한 번뿐입니다. 판정을 내리는 코드 자체를 먼저 검증해 보고 싶다면:

```bash
python3 skills/agent-coach/scripts/tests/run.py          # 테스트 219개, 표준 라이브러리만 사용
```

## golden-set-drafter

golden-set-drafter는 **골든셋이 아직 없을 때** 쓰는 동반 스킬입니다. agent-coach보다 먼저, agent-coach 대신 실행됩니다. 사용자가 주는 두 가지, 즉 대상 파일과 "무엇을 더 잘하게 만들고 싶은가"를 적은 발전 방향 문서로부터 골든셋 첫 버전(v1)의 초안을 만듭니다.

초안은 이렇게 만들어집니다:

- proposer(제안), adversary(공격), arbiter(판정)라는 AI 세 역할이 케이스 입력과 train 쪽 채점 기준을 놓고 서로 반박하며 다듬고, 합의에 이른 것만 남깁니다.
- 그다음 모든 train 케이스를 **실제 대상 파일 + 실제 운영 모델**로 한 번씩 돌려 봅니다. "지금 프롬프트가 이 문제를 정말 틀리는가"라는 증거를 케이스마다 붙여 두기 위해서입니다.
- 단 한 곳은 일부러 비워 둡니다: **held-out 채점 기준**입니다. 이 스킬은 그것만은 절대 쓰지 않습니다. 그것은 사용자가 씁니다. 채점 기준이 비어 있는 동안에는 agent-coach의 `op=split` 명령이 실행을 거부하므로, 개선 루프가 자신의 최종 시험을 "미리 공부"하는 일이 구조적으로 불가능합니다. 이 공란은 실수가 아니라 안전장치입니다.

### 사용법

> "`./agents/support-agent.md`를 개선하고 싶은데 골든셋이 아직 없어. 발전 방향은 `./direction.md`에 있어. 골든셋 초안부터 만들어줘."

스킬은 **초안**(held-out 채점 기준은 전부 공란)을 만들어 놓고 **그 지점에서 멈춥니다**. 다음 네 단계는 사용자의 몫입니다: ① held-out 입력을 검토하고, ② held-out 채점 기준을 전부 직접 쓰고, ③ `op=split` 명령을 실행하고(첫 실행은 실패하는 것이 정상이며, 그 거부가 곧 안전장치가 작동한다는 표시입니다), ④ 다 채운 뒤 다시 실행하면 골든셋이 동결됩니다(내용이 잠겨 그대로 확정됩니다). 그다음부터는 agent-coach가 이어받습니다.

## 저장소 구조

```
.
├── README.md · README.ko.md          # 이 랜딩 페이지 (영문 / 한국어)
├── docs/                             # 전체 매뉴얼 (영어 + 한국어 *.ko.md)
│   ├── agent-coach/                  #   golden-set · run-config · running · how-it-works
│   └── golden-set-drafter/           #   golden-set-drafter.md (drafter 문서)
├── examples/                         # 그대로 복사해 쓸 수 있는 예시, 스킬별 (en/ + ko/ 대칭)
│   ├── agent-coach/
│   └── golden-set-drafter/
├── skills/                           # Claude Code 스킬 본체 2개
│   ├── agent-coach/                  #   SKILL.md · agents/ · scripts/ (테스트 219개) · references/ · evals/ · assets/
│   └── golden-set-drafter/           #   SKILL.md · agents/ · scripts/ · references/ · evals/ · assets/
└── loop/                             # 실행별 작업 상태 (아래 설명)
```

각 실행의 작업 상태는 `loop/<대상 이름>/`에 남습니다. 전부 사람이 직접 열어 읽을 수 있는 일반 파일이고, 중단된 실행은 이 파일들로부터 이어서 재개할 수 있습니다:

- `golden-set.json`: 시험지(케이스 + 채점 기준)
- `prompt.current.md` / `prompt.candidate.md`: 루프가 편집하는 작업용 사본 (실제 파일이 아님)
- `history.jsonl`: 턴마다 한 줄씩, 무엇을 시도했고 점수가 어떻게 움직였는지
- `failure-log.jsonl`: 폐기되거나 중단된 시도와 그 이유
- `state.json`: 실행이 어디까지 진행됐는지 (완료한 단계를 반복하지 않고 재개하기 위한 기록)

## 요구 사항

- 번들 스크립트 실행에는 **Python 3.8 이상**이 필요합니다. **표준 라이브러리만** 사용하므로 따로 설치할 패키지는 없습니다.
- **Claude Code**(서브에이전트 지원)가 필요합니다. 실행·채점·제안·초안 네 역할을 각각 독립된 서브에이전트로, 깨끗한 컨텍스트에서 돌립니다. 역할 분리가 말뿐이 아니라 실제로 지켜지는 이유입니다.
